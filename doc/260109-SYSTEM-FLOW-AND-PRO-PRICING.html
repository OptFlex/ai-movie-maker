🔄 システムフロー全体図（DB + Storage アーキテクチャ）

📷 画像生成の流れ

1
🖥️
ユーザーがブラウザで操作
api-test-image.html で「画像を生成する」ボタンをクリック
プロンプト: "未来的な都市の風景、ネオンライト"
⬇️
2
⚡
Supabase Edge Function 呼び出し
POST /functions/v1/generate-image
フロントエンドから Edge Function にリクエスト
⬇️
3
🤖
Google Gemini API で画像生成
Edge Function が API キーを使って Google API を呼び出し
画像データ（Base64）を取得
⬇️
4
📤
フロントエンドに返却
Edge Function → ブラウザへ画像データ（Base64）を返す
ブラウザで画像をプレビュー表示
⬇️ 保存する場合
5
📦
Supabase Storage に画像を保存
storage.from('generated-images').upload()
パス例: user_abc123/20260108_123456.png
→ 公開URL取得可能
⬇️
6
🗄️
Supabase DB にメタデータ保存
supabase.from('generated_images').insert()
保存内容: プロンプト、Storage パス、品質、作成日時など
✅ 完了！

📜 履歴表示の流れ

1
🖥️
ユーザーが履歴ページを開く
history.html（仮）を開く
⬇️
2
🗄️
Supabase DB からメタデータ取得
supabase.from('generated_images').select('*')
→ 画像一覧（プロンプト、Storage パス、日時）を取得
高速！（画像本体は含まれない）
⬇️
3
📦
Storage の公開 URL を構築
storage.getPublicUrl(path)
各画像の Storage パスから公開 URL を生成
⬇️
4
🖼️
サムネイル一覧を表示
<img src="公開URL"> で画像を表示
ブラウザが Storage から画像を自動ダウンロード
CDN経由で高速配信！
✅ 履歴表示完了！

💡 なぜ DB + Storage が効率的なのか？

❌ DBのみの場合（非効率）

履歴表示のクエリ
SELECT * FROM generated_images LIMIT 100
⚠️ 問題点:
  • • 画像Base64（500KB）× 100枚 = 50MB転送
  • • クエリが遅い（数秒かかる）
  • • ブラウザのメモリを大量消費
データ構造
id: uuid
user_id: uuid
prompt: text
image_base64: text (500KB!)
created_at: timestamp
💔 1レコードが巨大（500KB）

✅ DB + Storage（効率的）

履歴表示のクエリ
SELECT * FROM generated_images LIMIT 100
✅ メリット:
  • • メタデータ（200B）× 100件 = 20KB転送
  • • クエリが高速（0.1秒以下）
  • • 画像は必要な時だけ読み込み
データ構造
id: uuid
user_id: uuid
prompt: text
storage_path: text (50B!)
created_at: timestamp
💚 1レコードが軽量（200B）
