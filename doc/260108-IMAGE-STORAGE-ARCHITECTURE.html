📊 画像履歴保存アーキテクチャ比較

Supabase DB + Storage vs DBのみ - 2026年1月8日作成

✅

アプローチ1: DB + Storage（推奨）

🖥️

フロントエンド

画像生成リクエスト
Base64画像データ受信
🗄️

Supabase DB

(メタデータのみ)

id: uuid
user_id: uuid
prompt: text
storage_path: text
quality: text
created_at: timestamp
サイズ: ~200 bytes/レコード
📦

Supabase Storage

(画像ファイル本体)

バケット: generated-images
{user_id}/
　└ 20260108_123456.png
サイズ: ~500KB - 5MB/画像

📥 保存の流れ

1. 画像生成
→
2. Storageに保存
→
3. DBにメタデータ保存
→
4. URL取得

✅ メリット

  • • 高速クエリ: メタデータのみ検索
  • • コスト効率: Storageは安価
  • • CDN配信: 画像を高速配信
  • • スケーラブル: 大量の画像に対応
  • • 動画対応: 将来的に動画も保存可能

⚠️ デメリット

  • • 実装が複雑: 2つのシステム管理
  • • 設定が必要: Storageバケット作成
  • • RLS設定: セキュリティルール2箇所

💻 実装コード例

// 1. Storageに画像を保存
const fileName = `${userId}/${Date.now()}.png`
const { data: uploadData } = await supabase.storage
  .from('generated-images')
  .upload(fileName, base64ToBlob(imageBase64))

// 2. DBにメタデータを保存
const { data: dbData } = await supabase
  .from('generated_images')
  .insert({
    user_id: userId,
    prompt: prompt,
    storage_path: fileName,
    quality: 'HD'
  })

// 3. 公開URLを取得
const { data: { publicUrl } } = supabase.storage
  .from('generated-images')
  .getPublicUrl(fileName)
⚡

アプローチ2: DBのみ（シンプル）

🖥️

フロントエンド

画像生成リクエスト
Base64画像データ受信
🗄️

Supabase DB

(メタデータ + 画像データ)

id: uuid
user_id: uuid
prompt: text
image_base64: text
quality: text
created_at: timestamp
サイズ: ~700KB - 7MB/レコード

📥 保存の流れ

1. 画像生成
→
2. DBに全て保存
→
3. 完了

✅ メリット

  • • シンプル: 1つのシステムのみ
  • • 設定不要: Storageの設定なし
  • • 実装が楽: コード量が少ない

⚠️ デメリット

  • • DBサイズ大: すぐに肥大化
  • • クエリ遅い: 大きなデータ転送
  • • コスト高: DB容量課金
  • • CDNなし: 画像配信が遅い
  • • 動画不可: 大きなファイル不向き

💻 実装コード例

// DBに全て保存（シンプル）
const { data } = await supabase
  .from('generated_images')
  .insert({
    user_id: userId,
    prompt: prompt,
    image_base64: imageBase64,  // Base64をそのまま保存
    quality: 'HD'
  })

💰 料金比較 (Supabase無料枠)

DB + Storage

DB容量制限
500 MB（メタデータのみなので余裕）
Storage容量制限
1 GB（画像約200-2000枚）
実質: 1000枚以上保存可能 ✅

DBのみ

DB容量制限
500 MB（全て保存）
Storage容量制限
使用しない
実質: 70-700枚で上限 ⚠️

🎯 結論

将来性とコストを考えると「DB + Storage」が圧倒的におすすめ！

  • ✅ 100枚以上保存する予定なら、DB + Storage一択
  • ✅ 将来的に動画も保存するなら、必須
  • ✅ CDN配信で高速表示
  • ⚡ プロトタイプやテストなら、DBのみでもOK
実装が少し複雑になりますが、長期的なメリットが大きいです
